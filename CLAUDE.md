# 코딩 가이드 (CODING COMPASS)

> 이 문서는 **모든 코딩 작업의 기준점**이다.
> 어떤 작업을 시작하든, 길을 잃었다고 느끼든, 반드시 이 문서로 돌아와서 현재 위치를 확인한다.
> Claude Code는 세션 시작 시 이 파일을 자동으로 읽으므로, 규칙은 항상 적용된다.

---

## 0. 최우선 원칙 (절대 규칙)

1. **요청받은 것만 한다.** 요청하지 않은 기능, 리팩토링, "김에 하는" 수정을 하지 않는다.
2. **이해 없이 코딩하지 않는다.** 기존 코드를 읽고 이해한 후에 수정한다.
3. **작게 나눠서 진행한다.** 한 번에 하나의 목표. 큰 작업은 단계로 쪼갠다.
4. **검증 없이 완료라고 하지 않는다.** 실행/테스트로 확인한 것만 "완료"다.
5. **막히면 멈추고 정리한다.** 같은 시도를 3번 반복했다면, 접근 방식 자체를 재검토한다.

---

## 1. 작업 시작 전 체크리스트

새 작업을 시작할 때 아래 질문에 먼저 답한다:

- [ ] **목표가 무엇인가?** 한 문장으로 말할 수 있어야 한다.
- [ ] **완료 조건은 무엇인가?** "무엇이 되면 끝인지" 명확히 한다.
- [ ] **어떤 파일을 건드리는가?** 수정 범위를 미리 정한다.
- [ ] **기존 코드/패턴이 있는가?** 새 파일을 만들기 전에 비슷한 파일이 이미 있는지 먼저 찾아보고, 있으면 재사용한다.
- [ ] **애매한 부분이 있는가?** 추측하지 말고 사용자에게 확인한다.

작업이 3단계 이상이면 **계획을 먼저 적고 시작한다** (TodoWrite 또는 문서로).

---

## 2. 표준 작업 흐름 (WORKFLOW)

모든 코딩은 아래 순서를 따른다. 단계를 건너뛰지 않는다.

```
① 이해 → ② 계획 → ③ 구현 → ④ 검증 → ⑤ 정리/커밋
```

### ① 이해 (Understand)
- 요구사항을 다시 읽는다. 애매하면 질문한다.
- 관련 코드를 검색하고(Grep/Glob) 읽는다(Read).
- 기존 구조·네이밍·패턴을 파악한다.

### ② 계획 (Plan)
- 수정할 파일과 순서를 나열한다.
- 큰 작업은 독립적으로 검증 가능한 작은 단계로 쪼갠다.
- 위험한 부분(삭제, 마이그레이션, 외부 API)을 미리 표시한다.

### ③ 구현 (Implement)
- 한 번에 한 단계만 구현한다.
- 기존 코드 스타일을 따른다 (들여쓰기, 네이밍, 주석 밀도, 라이브러리 선택).
- 새 의존성 추가는 최소화하고, 추가 시 이유를 남긴다.

### ④ 검증 (Verify)
- 코드를 실제로 실행하거나 테스트를 돌린다.
- 테스트가 없으면 최소한 해당 흐름을 수동으로 확인한다.
- 실패하면 원인을 파악하고 수정 → 재검증. **실패를 숨기지 않는다.**

### ⑤ 정리/커밋 (Commit)
- 임시 코드, 디버그 출력, 사용하지 않는 import를 제거한다.
- **동작하는 상태가 되면 그때그때 커밋한다.** 커밋은 "되돌아갈 수 있는 저장 지점"이다.
  문제가 생겨도 마지막 커밋으로 안전하게 돌아올 수 있다.
- 커밋 메시지는 "무엇을, 왜"를 담는다.
  (예: `로그인 버튼이 안 눌리던 버그 수정`, `메인 페이지에 검색 기능 추가`)
- 지정된 브랜치에만 push한다.

---

## 3. 코드 품질 원칙

- **가독성 > 영리함.** 읽는 사람이 바로 이해하는 코드를 쓴다.
- **KISS / YAGNI.** 지금 필요한 것만, 가장 단순한 방법으로 만든다.
- **DRY는 3번째부터.** 같은 코드가 3번 나오기 전에는 성급하게 추상화하지 않는다.
- **함수는 한 가지 일만.** 길어지면 쪼갠다.
- **에러는 삼키지 않는다.** 빈 catch 금지. 에러는 처리하거나 명확히 전달한다.
- **매직 넘버 금지.** 의미 있는 이름의 상수로 만든다.
- **주석은 "왜"를 설명.** 코드가 말하지 못하는 제약·이유만 주석으로 남긴다.

---

## 4. 안전 규칙 (하기 전에 반드시 확인)

| 행동 | 규칙 |
|---|---|
| 파일 삭제/덮어쓰기 | 대상 내용을 먼저 확인한다. 내가 만들지 않은 파일은 특히 주의 |
| DB/데이터 변경 | 되돌릴 방법(백업, 마이그레이션 롤백)을 먼저 확보한다 |
| 대규모 리팩토링 | 사용자 승인 없이 시작하지 않는다 |
| 시크릿/키 | 코드·커밋·로그에 절대 포함하지 않는다. `.env`는 커밋 금지 |
| force push | 원칙적으로 금지. 필요하면 `--force-with-lease`만, 사유와 함께 |
| 외부 배포/공개 행동 | 사용자 확인 후 진행한다 |

## 5. 길을 잃었을 때 (RECOVERY)

작업 중 방향을 잃었다고 느끼면 아래를 수행한다:

1. **멈춘다.** 코드를 더 쓰지 않는다.
2. **원래 목표를 다시 읽는다.** (1번 체크리스트의 "목표" 문장)
3. **현재 상태를 정리한다:** 완료된 것 / 진행 중인 것 / 남은 것.
4. **git 상태를 확인한다:** `git status`, `git diff`로 실제 변경분을 본다.
5. **탈선 여부를 판단한다:** 지금 하는 일이 목표에 직접 기여하는가?
   - 기여하지 않으면 → 그 작업을 버리거나(git checkout) 별도 작업으로 분리한다.
6. **같은 오류를 3번 이상 만났다면** → 접근 방식을 바꾸거나 사용자에게 상황을 보고한다.

---

## 6. 소통 규칙

- 진행 상황은 **결과부터** 보고한다 (무엇이 됐고, 무엇이 안 됐는지).
- 문제 발견 시 **즉시 공유**한다. 나중에 몰아서 말하지 않는다.
- 추측으로 진행한 부분이 있다면 **명시적으로 표시**한다.
- 요청 범위를 벗어나는 개선점을 발견하면 **제안만 하고 임의로 구현하지 않는다.**

## 7. 이 프로젝트에 대한 메모 (초보자용)

> 여기는 **"지금 만들고 있는 프로젝트가 뭔지" 적어두는 곳**이다.
> 처음에는 비워둬도 된다. 프로젝트를 시작하면 Claude에게
> **"7번 섹션 채워줘"** 라고 말하면 Claude가 알아서 채워준다.

쉽게 말하면 아래 3가지 질문에 대한 답을 적는 곳이다:

**① 이 프로젝트는 뭘로 만들었나?**
- (예: "파이썬으로 만든 웹사이트" / "자바스크립트로 만든 게임")
- 답: _아직 없음_

**② 어떻게 실행해서 눈으로 확인하나?**
- 코드를 고친 뒤 "잘 되는지 보는 방법"이다.
- (예: 터미널에 `npm run dev` 입력 → 브라우저에서 열어보기)
- 답: _아직 없음_

**③ 절대 건드리면 안 되는 것이 있나?**
- (예: "비밀번호 파일은 수정 금지" / "데이터 폴더는 삭제 금지")
- 답: _아직 없음_

> 💡 **왜 적어두나?** Claude는 매 세션 이 파일을 읽는다.
> 여기에 답이 적혀 있으면, 매번 다시 설명하지 않아도
> Claude가 프로젝트를 바로 이해하고 올바르게 실행·확인할 수 있다.

## 8. 세션 시작/종료 습관 (작업 기록)

세션이 바뀌어도 맥락이 끊기지 않도록, 매 세션 아래 습관을 지킨다.

### 세션을 시작할 때
1. `git status`로 저장소 상태를 확인한다 (커밋 안 된 변경이 있는지).
2. `docs/worklog.md`가 있으면 **먼저 읽고** 지난 작업이 어디까지 됐는지 파악한 뒤 시작한다.

### 세션을 마칠 때 (또는 긴 작업의 중간에)
`docs/worklog.md`에 날짜별로 기록한다. 다음 세션의 Claude가 이 기록을 보고 이어서 작업한다:

```markdown
## 2026-07-09
- 목표: ...
- 완료: ...
- 다음 할 일: ...
- 주의: ...
```

> 💡 초보자 팁: 이 기록도 직접 쓸 필요 없다. 작업을 마칠 때
> **"worklog 남겨줘"** 라고 말하면 Claude가 알아서 기록한다.
